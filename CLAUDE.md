# Dockbase ‚Äî Project Memory

## What is this?

A modular Docker deployment toolkit for multiple services on one Ubuntu 24.04 server. Stacks: WooCommerce shop (FrankenPHP + MariaDB + KeyDB + Matomo), Mailcow mail, and static website ‚Äî all behind a shared Caddy reverse proxy.

## Current phase: Multi-stack deployment

See `DOCKBASE-PLAN-FINAL.md` for the canonical spec.
See `docs/migration/` for legacy migration docs.

## Hard rules

- **Clean fixes only.** No volume hacks, no manual plugin installs, no workarounds.
- **All plugins via Composer.** Nothing lives only in a Docker volume.
- **Three config files.** Secrets in `config.env`, web routing in `config/domains.conf`, mail domains in `config/mail-domains.conf`.
- **Idempotent scripts.** Every script in `infra/` must be safely re-runnable.
- **No hardcoded domains.** Scripts read from config files, not inline strings.
- **No `steinmetz-mindelheim-shop.de`** anywhere in the implementation.

## Deploy modes

| Mode | Deploys |
|------|---------|
| `full` | shared infra + shop + mail + website |
| `shop` | shared infra + shop |
| `mail` | shared infra + mail |
| `web`  | shared infra + website |

## Key technical details

- Shared Caddy owns ports 80/443 and routes by Host header
- FrankenPHP listens on internal `:8080` only (no TLS)
- Mailcow web binds `127.0.0.1:8880/8443`, proxied by shared Caddy via `host.docker.internal`
- Mailcow is NOT on the `dockbase-proxy` Docker network
- CrowdSec reads shared Caddy logs at `/opt/dockbase/logs/caddy/`
- State tracking uses mode-prefixed entries: `full:harden`, `shop:setup-caddy`, etc.
- WordPress URLs in database require `wp search-replace` on domain change
- Backup tags: `shop-db`, `shop-files`, `shop-matomo`, `web-files`, `web-matomo`, `mailcow`
- Legacy tags (`db`, `uploads`, `languages`, `matomo`) supported for migration
- Backup runs daily at 04:30, after WP auto-update at 04:00, after reboot window at 03:00
